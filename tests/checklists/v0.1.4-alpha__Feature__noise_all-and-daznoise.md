## What this is

A human-runnable companion to v0.1.4-alpha. v0.1.4 adds three coupled
changes that need real-ComfyUI verification because mocks don't see the
LiteGraph UI, ComfyUI's `/object_info` cache, or the actual DazNoise
generators:

1. **DazNoise integration (Phase 3 of #14).** Five `DazNoise: *` entries
   appear in `noise_type_init`, `noise_type_sde`, and
   `noise_type_sde_substep` dropdowns when `dazzle-comfy-plasma-fast` is
   installed; absent when not.
2. **`noise_all` master selector (Phase 4 of #14).** Top-of-noise-group
   combo on `DazzleKSampler` that drives all three sub-widgets via JS
   splice-based hide/show. Force-synced server-side as defense in depth.
3. **Per-step-noise tooltip caveat (Phase 5 of #14, partial).** Tooltips
   on `noise_type_sde` / `noise_type_sde_substep` flag artifact-prone
   structured noise types.

This is the gating document before tagging `v0.1.4-alpha`.

## Prerequisites

1. **Working ComfyUI install** with `dazzle-ksampler` and (for full coverage) `smart-resolution-calc` custom nodes loaded.
2. **`dazzle-comfy-plasma-fast` installed** for HV.1, HV.3, Section 1.2, Section 4. HV.0 / Section 1.1 covers the absent-package path; if `dazzle-comfy-plasma-fast` is permanently installed, skip HV.0 and Section 1.1.
3. **A checkpoint** for sampling â€” model-agnostic; SDXL or Qwen-Image both fine.
4. **Frontend cache flush after restart**: hit Ctrl+Shift+R after every ComfyUI restart so the browser picks up the updated `/object_info` (new `noise_all` widget + DazNoise dropdown choices). Without this you'll see stale UI and tests will appear to fail spuriously.
5. **Reset between sections**: deleting and re-adding the `DazzleKSampler` node returns to a clean default-widget state. No env vars or shared dotfiles are involved (this is a frontend extension; nothing on the user's filesystem is touched).

### HV.1 â€” DazNoise types appear in dropdowns when the package is installed

### HV.2 â€” `noise_all` default hides sub-widgets; `custom` shows them

Validates: the JS splice pattern, `noise_all` default value, server-side force-sync.

- [ ] **Steps:**
  1. Add a fresh `DazzleKSampler` node.
  2. Observe initial widget layout.
  3. Set `noise_all = custom`.
  4. Set `noise_all = pink`.
- [ ] **Expected:**
  - Step 2: `noise_all` is visible and reads `gaussian`. The three `noise_type_*` sub-widgets are NOT visible.
  - Step 3: The three sub-widgets appear directly below `noise_all`, each defaulting to `gaussian`.
  - Step 4: The three sub-widgets disappear.
- [ ] **Why this matters:** This is the headline UX of Phase 4. If the splice pattern is broken, users see a cluttered node always â€” defeats the point of the master selector.

### HV.3 â€” DazNoise per-step injection doesn't crash the sampler

Validates: adapter normalization on the per-step path, `resolve_noise_class()` in `rk_noise_sampler_beta.py`, the RGBâ†’grayscaleâ†’broadcastâ†’normalize pipeline.

- [ ] **Steps:**
  1. DazzleKSampler with `noise_all = DazNoise: Pink`, fixed seed, any model + prompt.

- [ ] **Expected:** All three runs complete without exception. Each produces a different image (Plasma should show visible halftone-style structure â€” that's expected behavior, not a bug; the tooltip warns about it).

### HV.4 â€” #12-class regression: wire-input + widget change does not break hide logic

Validates: the new JS splice pattern doesn't repeat v0.1.2's mistake of mutating `widget.type` mid-flight.

- [ ] **Steps:**

  2. Wire SmartResCalc's `seed` output to DazzleKSampler's `seed` input.

  4. Set `noise_all = custom`, then back to `brown`.
  5. Disconnect the seed wire.
- [ ] **Expected:** Through every step, no widget vanishes unexpectedly. Sub-widgets show/hide correctly on `noise_all` changes. Seed widget restores cleanly on disconnect.
- [ ] **Why this matters:** Issue #12 is the bug we just spent v0.1.3 fixing. v0.1.4 reintroduces hide-logic JS â€” it absolutely must not regress this class.

### HV.5 â€” Workflow save/load round-trip preserves hidden sub-widget values

Validates: ComfyUI's positional widget serialization sees all widgets we added (including the hidden ones), and noise_all value survives reload.

- [ ] **Steps:**
  1. DazzleKSampler with `noise_all = custom`, `noise_type_init = brown`, `noise_type_sde = pink`, `noise_type_sde_substep = violet`. Save the workflow JSON.
  2. Set `noise_all = blue` (sub-widgets hide). Save again as a separate workflow.
  3. Clear ComfyUI graph. Load workflow #1.
  4. Clear again. Load workflow #2. Then set `noise_all = custom`.
- [ ] **Expected:**
  - Workflow #1 reload: `noise_all = custom`, sub-widgets show with `brown / pink / violet`.

- [ ] **Why this matters:** If save/load doesn't round-trip, users lose configuration silently. ComfyUI serializes by position, not name, so the widget order in INPUT_TYPES vs the runtime widgets array must stay aligned even when the JS layer hides widgets.

## Section 3: Server-side force-sync (defense in depth)

Validates the Python-side reassignment in `DazzleKSampler.main()` so behavior is correct even if the JS layer is bypassed.

### 3.1 Force-sync overrides hidden sub-widget values

- [ ] **Steps:**
  1. DazzleKSampler with `noise_all = custom`. Set sub-widgets `brown / pink / violet`. Generate. Capture output.
  2. Set `noise_all = blue` (sub-widgets hide). Generate.
  3. Set `noise_all = custom`. Generate (sub-widgets still `brown / pink / violet`).
- [ ] **Expected:**
  - Step 1 and step 3 outputs match (same sub-widget values, same seed).
  - Step 2 output differs from both (force-sync replaced the hidden values with `blue`).
- [ ] **Why:** If the force-sync is buggy, hidden sub-widget values would leak through and `noise_all` would be a UI-only lie.

### 3.2 Force-sync via wire-input on `noise_all` (advanced)

- [ ] **Steps:**
  1. Convert `noise_all` to an input slot (right-click â†’ "Convert noise_all to input").
  2. Wire any STRING source that supplies the literal `"pink"`.
  3. Generate.
- [ ] **Expected:** No crash. Output reflects per-step `pink` injection. The hidden sub-widget values (whatever they were) do not leak through.
- [ ] **Note:** This path is rarely exercised by users; mark MANUAL if wiring a STRING into a combo input isn't supported in your ComfyUI version.

## What the automated tests DON'T cover

- **ComfyUI's `/object_info` cache surfacing the new dropdown choices** â€” only the live frontend after Ctrl+Shift+R proves this works.
- **JS splice pattern actually showing/hiding widgets** in real LiteGraph â€” unit tests stub `NODE_CLASS_MAPPINGS` but never spin up a browser.
- **Widget callback chaining surviving wire-input conversion** (the #12 class) â€” requires real ComfyUI + LiteGraph DOM behavior.
- **`noise_all = custom` honoring sub-widget values across save/load** â€” depends on ComfyUI's positional serialization, which mocks bypass.
- **DazNoise types producing recognizable structured noise in real generations** â€” the adapter normalizes to mean=0, std=1, which can make stub-generator output look ~Gaussian even when the real generator produces clear plasma structure. Only real-model generation can confirm visible structure.
- **The actual `dazzle-comfy-plasma-fast` generators** â€” we stub `JDC_OmniNoise` / `JDC_Plasma` / etc. with deterministic `torch.rand` in unit tests. The real generators may have edge cases (e.g. very large image sizes, specific aspect ratios) that the stub doesn't exercise.
- **Tooltip rendering in the actual UI** â€” string content is in code, but visual rendering, line wrapping, and click-to-dismiss behavior are LiteGraph-side.
- **Performance feel** â€” 5-DazNoise-type dropdowns are tiny, but verify the hide/show transition isn't visibly janky.
